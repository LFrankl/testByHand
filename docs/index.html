TCP 自定义协议 网络编程

在 TCP 之上设计带定长头部的应用层协议，解决粘包 / 拆包问题。

设计要点

  • 12 字节定长头部：Magic(2) + Version(1) + Type(1) + Reserved(1) + Seq(4) + BodyLen(4)
  • 粘包解决方案：io.ReadFull 先读固定 12 字节头，再按 BodyLen 精确读 Body，永不多读
  • bufio.Reader 包装连接：减少系统调用，批量读取
  • 校验 Magic + Version：防止脏数据或版本不兼容
  • 大端序（binary.BigEndian）：网络字节序的惯例
  • atomic.AddUint32 生成单调递增序列号，支持并发客户端请求匹配
tcp-protocol/answer/main.go
协议格式：定长头部是解决 TCP 粘包最简洁的方案。接收方先 io.ReadFull(r, headerBuf) 读满 12 字节，解析出 BodyLen，再 io.ReadFull(r, body) 精确读取，完全不依赖 delimiter。

Encode/Decode 对称性：binary.Write 按大端序把整个 Header struct 写入 buffer，binary.Read 反向还原——字段顺序必须与内存布局完全对齐，否则会解析错位。

服务端模型：Accept loop + go handleConn(conn)，每连接一个 goroutine，bufio.Reader 减少小读的 syscall 次数。processMessage 基于 Type 路由，返回的 Seq 与请求一致，便于多路复用场景下的请求匹配。

核心考察点

问题答案要点
TCP 粘包是什么？TCP 是流式协议，内核可能把多个 Write 合并或把一个 Write 拆分发送。应用层必须自己定界。
有哪些解帧方案？① 定长头部含 BodyLen（本题）② 固定分隔符（如 HTTP 的 \r\n）③ 固定长度消息
为何用 io.ReadFull？普通 Read 可能返回少于请求字节数；ReadFull 循环直到读满或 EOF/error，保证原子性。
大端 vs 小端？网络协议惯用大端（Network byte order）；x86 CPU 是小端。Go 的 binary.BigEndian 负责转换。

LRU 缓存 数据结构

线程安全的 LRU（最近最少使用）缓存，Get/Put 均 O(1)。

设计要点

  • 双向链表 + HashMap：链表维护访问顺序（头=最近，尾=最旧），Map 实现 O(1) 定位节点
  • 哨兵节点：head/tail 永不存数据，消除边界判空，所有链表操作统一无分支
  • moveToFront = remove + addToFront：复用两个基础操作，代码最小化
  • sync.Mutex 保护整个 cache map 和链表，所有公开方法加锁
  • 淘汰时从 tail.prev 摘除，同时 delete(cache, key) 保持 Map 与链表一致
lru/answer/lru.go
为何是 O(1)：Get → Map 查找 O(1) → 链表 remove+addToFront O(1)（直接操作指针，不遍历）。Put → Map 查找/插入 O(1) → 链表操作 O(1)。

哨兵节点的价值：没有哨兵时，插入第一个节点需要判断 head==nil；删除最后节点需要判断 tail==nil。哨兵使得 addToFront 和 removeLast 的代码路径完全统一，减少 bug 表面积。

并发安全：Get 和 Put 都修改链表顺序（Get 也要 moveToFront），因此两者都需要写锁。若读操作比写多得多，可考虑读写分离，但实际上 LRU 的 Get 会修改状态，用 RWMutex 反而复杂化。

核心考察点

问题答案要点
为何需要双向链表？单向链表删除任意节点需要找前驱节点（O(n)）；双向链表通过 n.prev 直接拿到前驱，O(1) 删除。
哨兵节点有什么好处？统一边界处理，消除"链表为空时特殊判断"，所有 add/remove 操作代码路径一致。
Get 为何也要加写锁？Get 后需要 moveToFront（修改链表），所以不能用只读锁。
如何支持泛型 key/value？Go 1.18+ 可用类型参数 [K comparable, V any] 替换 int。

秒杀服务 并发 · HTTP

高并发秒杀核心逻辑 + HTTP 接口，防超卖、防重复购买。

设计要点

  • 原子临界区：查重复 → 查库存 → 扣库存 → 写订单，四步必须在同一把锁内完成，缺一不可
  • sold map[int64]bool：O(1) 用户去重，先于库存检查（短路优化）
  • 超卖根因：若"查库存"和"扣库存"分两次加锁，两次锁之间库存可能被他人耗尽
  • HTTP 层用 errors.Is 区分业务错误，统一返回 409 Conflict
  • atomic.AddUint64 生成订单序列号，无需额外锁
seckill/answer/seckill.go
超卖的产生与防止：假设库存剩 1，两个 goroutine 同时进入。若分两步加锁：两者都读到 stock=1，都通过检查，先后扣减变成 -1。本实现将"读+写"置于同一把锁，第二个 goroutine 进入时 stock 已为 0，直接返回 ErrOutOfStock。

操作顺序：先查 sold（去重）再查库存。因为重复购买在高并发下更常见，早返回节省 CPU；若反过来，用户 A 秒杀成功后再次请求，需要等到库存检查后才能去重，逻辑无误但略低效。
seckill/answer/handler.go
HTTP 设计：Handler 实现 http.Handler 接口而非注册全局路由，便于测试（直接 httptest.NewServer(handler)）。

状态码语义：409 Conflict 表示"请求与当前服务器状态冲突"，比 400 Bad Request 更准确地描述超卖/重购的语义。errors.Is 保证 wrap 后的错误也能正确匹配。

核心考察点

问题答案要点
什么是超卖？如何防止？多个并发请求都通过库存检查但库存不足时发生。防止方法：在同一把锁内完成检查+扣减，或用数据库原子 UPDATE + 行锁。
为何不用 atomic 替代 Mutex？需要同时保护三个字段（stock、sold、orders）的一致性，atomic 只能保护单个变量，无法保证组合操作的原子性。
生产环境如何扩展？Redis DECR 原子扣减库存（分布式）+ Kafka 异步落单，避免 DB 热点。Lua 脚本保证 Redis 上的 check-and-decrement 原子性。

Web 框架 框架设计

基于 net/http 的 Web 框架：Trie 路由树 + 洋葱中间件 + 分组。

设计要点

  • Trie 路由：每个 HTTP Method 一棵树，支持 :param（命名参数）和 *wildcard（通配）
  • 洋葱模型：中间件和 handler 存入同一个 []HandlerFunc，c.Next() 推进 index，c.Abort() 跳过剩余
  • 分组：RouterGroup 共享同一 Engine；addRoute 时遍历所有 group，前缀匹配则收集其中间件
  • Context 封装请求全部状态：Writer、Request、Params、keys(KV存储)
  • 内置 Logger（洋葱后置统计耗时）和 Recovery（捕获 panic 返回 500）
webframe/answer/context.go
中间件机制：handlers []HandlerFunc 存放中间件链，index 从 -1 开始。Next() 递增 index 并执行对应 handler，中间件可以在 c.Next() 前后插入前置/后置逻辑（洋葱）。Abort() 将 index 设为 len，后续循环条件不成立，链路终止。

KV 存储：用 sync.RWMutex 保护 keys map，允许中间件向 handler 传递数据（如认证后存入 userID）。
webframe/answer/router.go
Trie 路由：路径按 / 分段，每段对应一个 node。isWild=true 的节点（以 : 或 * 开头）能匹配任意同位段。

insert vs search 的区别：insert 用 matchChild（找第一个匹配）——精确段优先，保证静态路由不被参数路由覆盖；search 用 matchChildren（找所有匹配）——需回溯以防歧义路由。

参数提取：匹配成功后，将模式路径各段（如 :id）与实际 URL 各段对齐，提取具名参数写入 params map。
webframe/answer/engine.go
分组中间件：addRoute 遍历 engine.groups，凡是当前完整路径以某 group prefix 开头的，该 group 的中间件就适用。这样 /api/v1/users 会自动收集 root group 和 /api group 的中间件。

共享 Engine：所有 RouterGroup 指向同一个 Engine 指针，共享 router 和 groups 列表。这是组合模式，避免了复杂的树形父子关系。

核心考察点

问题答案要点
Trie 路由 vs map 路由？map 只支持精确路由；Trie 支持动态参数（:id）和通配（*filepath），查找复杂度 O(路径段数)。
洋葱模型如何实现前置/后置？中间件在 c.Next() 之前的代码是前置，之后的是后置。Next() 递归推进使调用栈天然形成洋葱层级。
Abort 是否立即停止？是。将 index 跳到末尾，Next() 的 for 循环条件不满足，后续 handler 不再执行。但 Abort 之前的中间件的 defer 仍然会执行。

工作池 并发原语

固定大小 goroutine 池，含 Future 模式与 FanIn/FanOut Pipeline。

设计要点（两大经典陷阱）

  • 陷阱1 — 关闭 channel 引发 panic：用 close(taskQueue) 通知退出时，Submit 可能在已关闭的 channel 上写入，panic。解法：taskQueue 永不关闭，用 nil sentinel 通知 worker 退出
  • 陷阱2 — wg.Add 位置错误导致 Shutdown 提前返回：若 worker 内部才 Add(1)，Shutdown 调用 Wait() 时队列中可能还有任务但 wg 计数已为 0。解法：wg.Add(1) 在 Submit 时、持锁期间执行
  • mu 保护 shutdown 检查与 wg.Add 原子性，防止 Shutdown 与 Submit 竞态
  • Shutdown 顺序：标记关闭 → wg.Wait() → 发 nil sentinel，确保任务处理完才退出 worker
  • Future: channel 缓冲为 1，worker 写入结果不阻塞（即使调用方从未 Get）
workerpool/answer/pool.go
nil sentinel 退出：worker 的 for task := range taskQueue 会阻塞等待，收到 nil 后 return。Shutdown 发送 numWorkers 个 nil，精确唤醒每个 worker 各一次。

Shutdown 为何先 Wait 再发 sentinel：Wait 返回时 taskQueue 已经被 worker 清空（所有任务执行完），此时发 nil，worker 读到 nil 时队列保证为空，干净退出。若先发 sentinel，nil 可能被某个 worker 在处理完真实任务前就读走，其他任务永远阻塞在 taskQueue 中。

FanOut 精髓：多个 goroutine 并发 range in，Go channel 天然支持多读者竞争，无需额外分发逻辑——这是 fan-out 模式最简洁的表达。

核心考察点

问题答案要点
为何 wg.Add 不放在 worker 内部？Worker 取到任务后才 Add，而 Shutdown 的 wg.Wait() 可能在 Add 前就执行，计数为 0，Wait 立即返回，导致部分任务未完成就退出。
为何不用 close(taskQueue)？关闭后若 Submit 仍写入会 panic；且多个 goroutine 同时写一个已关闭的 channel 是 fatal error。nil sentinel 只需发送 N 次，精确、安全。
Future 的 ch 为何容量为 1？让 worker 能无阻塞写入结果。若容量为 0，worker 必须等调用方 Get，造成 worker 白白占用（goroutine leak 风险）。
FanIn 如何安全关闭 out？每个转发 goroutine 执行 wg.Done()，一个专用 goroutine wg.Wait() 后 close(out)，确保所有数据转发完毕才关闭。

连接池 资源管理

懒加载连接池：idle channel 兼做队列和信号量，CAS 无锁新建，优雅关闭。

设计要点

  • idle channel 双重角色：len(idle) = 空闲数，cap(idle) = maxSize；阻塞读等空闲连接，非阻塞写归还
  • CAS 控制新建：atomic.CompareAndSwapInt64(&total, cur, cur+1) 保证并发时只有一个 goroutine 成功新建，避免超量
  • 惰性淘汰：Get 时检查健康状态和空闲超时，不健康则 total-- 后 continue；后台无定时清理 goroutine
  • inUseDone WaitGroup：Get 成功时 Add(1)，Put 时 Done()；Close 调用 Wait() 等全部归还
  • closeCh chan struct{}：close 广播给所有阻塞的 Get，无需逐个通知
connpool/answer/conn.go
模拟连接对象，持有健康标志和最后使用时间。MarkUnhealthy() 模拟连接断开，Do() 模拟执行查询（不健康时返回 error）。
connpool/answer/pool.go
Get 三步逻辑：① 非阻塞取 idle（有空闲连接立即返回）→ ② idle 空则 CAS 新建（total 未达上限）→ ③ 达到上限则阻塞 select 等 idle/closeCh/ctx。每步都检查健康和超时，不合格则 total-- 重新循环。

CAS 的必要性：若用 mutex 保护 total 的读-改-写，则每次 Get 都需要加锁，高并发下成为瓶颈。CAS 失败代价小（重新循环），在连接数未满时几乎总是成功一次。

Close 顺序严格有序：① atomic.Store(closed, 1)（后续 Get 直接返回错误）→ ② close(closeCh)（唤醒阻塞的 Get）→ ③ inUseDone.Wait()（等借出连接归还）→ ④ 排空 idle。顺序不能乱，否则可能泄漏连接或 double-close。

核心考察点

问题答案要点
idle channel 为何能当信号量？cap(idle)=maxSize，写满后阻塞相当于 semaphore；但这里 total 独立计数，channel 只放空闲连接，借出的不在 channel 里。
为何 Put 用非阻塞 select+default？极端情况（如 Close 后仍有连接归还），idle 已满则 Put 阻塞。non-blocking 防止归还方永久阻塞。
如何防止 Close 时泄漏借出的连接？inUseDone WaitGroup：每次 Get 成功 Add(1)，Put 时 Done()；Close 调用 Wait() 阻塞直到全部归还。
健康检查在哪里做？惰性：取出时检查（Get）和归还时检查（Put），不健康直接丢弃并 total--。无后台 goroutine，简单可靠。

限流器 流量控制

令牌桶（惰性补充）+ 滑动窗口（时间戳队列），两种经典限流算法完整实现。

设计要点

  • 令牌桶 — 惰性补充：不起后台 goroutine，每次 Allow/Wait 时按"经过时间 × rate"计算补充量，min(tokens+补充, burst)
  • Allow 非阻塞；Wait 计算需要等待的时长，time.After + ctx 双路 select
  • 滑动窗口 — 时间戳队列：维护升序时间戳切片，每次 Allow 先淘汰窗口外记录，再判总数是否达到 limit
  • 滑动窗口空间复杂度 O(limit)：超过 limit 的请求被拒，不追加时间戳，无内存泄漏
  • 两者都用 sync.Mutex，全程持锁，并发安全
ratelimiter/answer/ratelimiter.go
令牌桶惰性补充的优势：无后台 goroutine，无需 Stop/Close，生命周期简单。代价是每次操作多一次时间计算（纳秒级，可忽略）。Go 标准库 golang.org/x/time/rate 也采用这种设计。

Wait 的原子性：先释放锁计算等待时长，等待结束后再次加锁 refill+扣令牌。两次加锁之间令牌可能被其他 goroutine 消耗，所以等待结束后必须再次 refill，确保令牌充足。

滑动窗口 vs 固定窗口：固定窗口（计数器在整点重置）在窗口边界有"突刺"问题——前一窗口末尾 + 后一窗口开头可以瞬间通过 2×limit 请求。滑动窗口以"当前时刻往前推 windowSize"为视口，无突刺，但内存占用更高。

核心考察点

问题答案要点
令牌桶 vs 漏桶？令牌桶允许突发（桶满时可消耗 burst 个令牌）；漏桶以恒定速率流出，无突发能力。令牌桶更适合允许短暂流量峰值的场景。
滑动窗口的时间复杂度？O(k)，k 为窗口内请求数。淘汰过期记录是线性扫描；若用二分搜索可优化到 O(log k)，但窗口通常不大，线性已足够。
为何滑动窗口空间是 O(limit)？超过 limit 的请求被拒绝不追加，时间戳切片最多 limit 个元素；旧时间戳被淘汰后切片缩减，内存稳定。
生产中用哪个？单机限流优先令牌桶（x/time/rate）；分布式限流用 Redis INCR + expire（固定窗口）或 Redis sorted set（滑动窗口）。
