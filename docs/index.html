TCP 自定义协议 网络编程

在 TCP 之上设计带定长头部的应用层协议，解决粘包 / 拆包问题。

设计要点

  • 12 字节定长头部：Magic(2) + Version(1) + Type(1) + Reserved(1) + Seq(4) + BodyLen(4)
  • 粘包解决方案：io.ReadFull 先读固定 12 字节头，再按 BodyLen 精确读 Body，永不多读
  • bufio.Reader 包装连接：减少系统调用，批量读取
  • 校验 Magic + Version：防止脏数据或版本不兼容
  • 大端序（binary.BigEndian）：网络字节序的惯例
  • atomic.AddUint32 生成单调递增序列号，支持并发客户端请求匹配
tcp-protocol/answer/main.go
协议格式：定长头部是解决 TCP 粘包最简洁的方案。接收方先 io.ReadFull(r, headerBuf) 读满 12 字节，解析出 BodyLen，再 io.ReadFull(r, body) 精确读取，完全不依赖 delimiter。

Encode/Decode 对称性：binary.Write 按大端序把整个 Header struct 写入 buffer，binary.Read 反向还原——字段顺序必须与内存布局完全对齐，否则会解析错位。

服务端模型：Accept loop + go handleConn(conn)，每连接一个 goroutine，bufio.Reader 减少小读的 syscall 次数。processMessage 基于 Type 路由，返回的 Seq 与请求一致，便于多路复用场景下的请求匹配。

核心考察点

问题答案要点
TCP 粘包是什么？TCP 是流式协议，内核可能把多个 Write 合并或把一个 Write 拆分发送。应用层必须自己定界。
有哪些解帧方案？① 定长头部含 BodyLen（本题）② 固定分隔符（如 HTTP 的 \r\n）③ 固定长度消息
为何用 io.ReadFull？普通 Read 可能返回少于请求字节数；ReadFull 循环直到读满或 EOF/error，保证原子性。
大端 vs 小端？网络协议惯用大端（Network byte order）；x86 CPU 是小端。Go 的 binary.BigEndian 负责转换。

LRU 缓存 数据结构

线程安全的 LRU（最近最少使用）缓存，Get/Put 均 O(1)。

设计要点

  • 双向链表 + HashMap：链表维护访问顺序（头=最近，尾=最旧），Map 实现 O(1) 定位节点
  • 哨兵节点：head/tail 永不存数据，消除边界判空，所有链表操作统一无分支
  • moveToFront = remove + addToFront：复用两个基础操作，代码最小化
  • sync.Mutex 保护整个 cache map 和链表，所有公开方法加锁
  • 淘汰时从 tail.prev 摘除，同时 delete(cache, key) 保持 Map 与链表一致
lru/answer/lru.go
为何是 O(1)：Get → Map 查找 O(1) → 链表 remove+addToFront O(1)（直接操作指针，不遍历）。Put → Map 查找/插入 O(1) → 链表操作 O(1)。

哨兵节点的价值：没有哨兵时，插入第一个节点需要判断 head==nil；删除最后节点需要判断 tail==nil。哨兵使得 addToFront 和 removeLast 的代码路径完全统一，减少 bug 表面积。

并发安全：Get 和 Put 都修改链表顺序（Get 也要 moveToFront），因此两者都需要写锁。若读操作比写多得多，可考虑读写分离，但实际上 LRU 的 Get 会修改状态，用 RWMutex 反而复杂化。

核心考察点

问题答案要点
为何需要双向链表？单向链表删除任意节点需要找前驱节点（O(n)）；双向链表通过 n.prev 直接拿到前驱，O(1) 删除。
哨兵节点有什么好处？统一边界处理，消除"链表为空时特殊判断"，所有 add/remove 操作代码路径一致。
Get 为何也要加写锁？Get 后需要 moveToFront（修改链表），所以不能用只读锁。
如何支持泛型 key/value？Go 1.18+ 可用类型参数 [K comparable, V any] 替换 int。

秒杀服务 并发 · HTTP

高并发秒杀核心逻辑 + HTTP 接口，防超卖、防重复购买。

设计要点

  • 原子临界区：查重复 → 查库存 → 扣库存 → 写订单，四步必须在同一把锁内完成，缺一不可
  • sold map[int64]bool：O(1) 用户去重，先于库存检查（短路优化）
  • 超卖根因：若"查库存"和"扣库存"分两次加锁，两次锁之间库存可能被他人耗尽
  • HTTP 层用 errors.Is 区分业务错误，统一返回 409 Conflict
  • atomic.AddUint64 生成订单序列号，无需额外锁
seckill/answer/seckill.go
超卖的产生与防止：假设库存剩 1，两个 goroutine 同时进入。若分两步加锁：两者都读到 stock=1，都通过检查，先后扣减变成 -1。本实现将"读+写"置于同一把锁，第二个 goroutine 进入时 stock 已为 0，直接返回 ErrOutOfStock。

操作顺序：先查 sold（去重）再查库存。因为重复购买在高并发下更常见，早返回节省 CPU；若反过来，用户 A 秒杀成功后再次请求，需要等到库存检查后才能去重，逻辑无误但略低效。
seckill/answer/handler.go
HTTP 设计：Handler 实现 http.Handler 接口而非注册全局路由，便于测试（直接 httptest.NewServer(handler)）。

状态码语义：409 Conflict 表示"请求与当前服务器状态冲突"，比 400 Bad Request 更准确地描述超卖/重购的语义。errors.Is 保证 wrap 后的错误也能正确匹配。

核心考察点

问题答案要点
什么是超卖？如何防止？多个并发请求都通过库存检查但库存不足时发生。防止方法：在同一把锁内完成检查+扣减，或用数据库原子 UPDATE + 行锁。
为何不用 atomic 替代 Mutex？需要同时保护三个字段（stock、sold、orders）的一致性，atomic 只能保护单个变量，无法保证组合操作的原子性。
生产环境如何扩展？Redis DECR 原子扣减库存（分布式）+ Kafka 异步落单，避免 DB 热点。Lua 脚本保证 Redis 上的 check-and-decrement 原子性。

Web 框架 框架设计

基于 net/http 的 Web 框架：Trie 路由树 + 洋葱中间件 + 分组。

设计要点

  • Trie 路由：每个 HTTP Method 一棵树，支持 :param（命名参数）和 *wildcard（通配）
  • 洋葱模型：中间件和 handler 存入同一个 []HandlerFunc，c.Next() 推进 index，c.Abort() 跳过剩余
  • 分组：RouterGroup 共享同一 Engine；addRoute 时遍历所有 group，前缀匹配则收集其中间件
  • Context 封装请求全部状态：Writer、Request、Params、keys(KV存储)
  • 内置 Logger（洋葱后置统计耗时）和 Recovery（捕获 panic 返回 500）
webframe/answer/context.go
中间件机制：handlers []HandlerFunc 存放中间件链，index 从 -1 开始。Next() 递增 index 并执行对应 handler，中间件可以在 c.Next() 前后插入前置/后置逻辑（洋葱）。Abort() 将 index 设为 len，后续循环条件不成立，链路终止。

KV 存储：用 sync.RWMutex 保护 keys map，允许中间件向 handler 传递数据（如认证后存入 userID）。
webframe/answer/router.go
Trie 路由：路径按 / 分段，每段对应一个 node。isWild=true 的节点（以 : 或 * 开头）能匹配任意同位段。

insert vs search 的区别：insert 用 matchChild（找第一个匹配）——精确段优先，保证静态路由不被参数路由覆盖；search 用 matchChildren（找所有匹配）——需回溯以防歧义路由。

参数提取：匹配成功后，将模式路径各段（如 :id）与实际 URL 各段对齐，提取具名参数写入 params map。
webframe/answer/engine.go
分组中间件：addRoute 遍历 engine.groups，凡是当前完整路径以某 group prefix 开头的，该 group 的中间件就适用。这样 /api/v1/users 会自动收集 root group 和 /api group 的中间件。

共享 Engine：所有 RouterGroup 指向同一个 Engine 指针，共享 router 和 groups 列表。这是组合模式，避免了复杂的树形父子关系。

核心考察点

问题答案要点
Trie 路由 vs map 路由？map 只支持精确路由；Trie 支持动态参数（:id）和通配（*filepath），查找复杂度 O(路径段数)。
洋葱模型如何实现前置/后置？中间件在 c.Next() 之前的代码是前置，之后的是后置。Next() 递归推进使调用栈天然形成洋葱层级。
Abort 是否立即停止？是。将 index 跳到末尾，Next() 的 for 循环条件不满足，后续 handler 不再执行。但 Abort 之前的中间件的 defer 仍然会执行。

工作池 并发原语

固定大小 goroutine 池，含 Future 模式与 FanIn/FanOut Pipeline。

设计要点（两大经典陷阱）

  • 陷阱1 — 关闭 channel 引发 panic：用 close(taskQueue) 通知退出时，Submit 可能在已关闭的 channel 上写入，panic。解法：taskQueue 永不关闭，用 nil sentinel 通知 worker 退出
  • 陷阱2 — wg.Add 位置错误导致 Shutdown 提前返回：若 worker 内部才 Add(1)，Shutdown 调用 Wait() 时队列中可能还有任务但 wg 计数已为 0。解法：wg.Add(1) 在 Submit 时、持锁期间执行
  • mu 保护 shutdown 检查与 wg.Add 原子性，防止 Shutdown 与 Submit 竞态
  • Shutdown 顺序：标记关闭 → wg.Wait() → 发 nil sentinel，确保任务处理完才退出 worker
  • Future: channel 缓冲为 1，worker 写入结果不阻塞（即使调用方从未 Get）
workerpool/answer/pool.go
nil sentinel 退出：worker 的 for task := range taskQueue 会阻塞等待，收到 nil 后 return。Shutdown 发送 numWorkers 个 nil，精确唤醒每个 worker 各一次。

Shutdown 为何先 Wait 再发 sentinel：Wait 返回时 taskQueue 已经被 worker 清空（所有任务执行完），此时发 nil，worker 读到 nil 时队列保证为空，干净退出。若先发 sentinel，nil 可能被某个 worker 在处理完真实任务前就读走，其他任务永远阻塞在 taskQueue 中。

FanOut 精髓：多个 goroutine 并发 range in，Go channel 天然支持多读者竞争，无需额外分发逻辑——这是 fan-out 模式最简洁的表达。

核心考察点

问题答案要点
为何 wg.Add 不放在 worker 内部？Worker 取到任务后才 Add，而 Shutdown 的 wg.Wait() 可能在 Add 前就执行，计数为 0，Wait 立即返回，导致部分任务未完成就退出。
为何不用 close(taskQueue)？关闭后若 Submit 仍写入会 panic；且多个 goroutine 同时写一个已关闭的 channel 是 fatal error。nil sentinel 只需发送 N 次，精确、安全。
Future 的 ch 为何容量为 1？让 worker 能无阻塞写入结果。若容量为 0，worker 必须等调用方 Get，造成 worker 白白占用（goroutine leak 风险）。
FanIn 如何安全关闭 out？每个转发 goroutine 执行 wg.Done()，一个专用 goroutine wg.Wait() 后 close(out)，确保所有数据转发完毕才关闭。

连接池 资源管理

懒加载连接池：idle channel 兼做队列和信号量，CAS 无锁新建，优雅关闭。

设计要点

  • idle channel 双重角色：len(idle) = 空闲数，cap(idle) = maxSize；阻塞读等空闲连接，非阻塞写归还
  • CAS 控制新建：atomic.CompareAndSwapInt64(&total, cur, cur+1) 保证并发时只有一个 goroutine 成功新建，避免超量
  • 惰性淘汰：Get 时检查健康状态和空闲超时，不健康则 total-- 后 continue；后台无定时清理 goroutine
  • inUseDone WaitGroup：Get 成功时 Add(1)，Put 时 Done()；Close 调用 Wait() 等全部归还
  • closeCh chan struct{}：close 广播给所有阻塞的 Get，无需逐个通知
connpool/answer/conn.go
模拟连接对象，持有健康标志和最后使用时间。MarkUnhealthy() 模拟连接断开，Do() 模拟执行查询（不健康时返回 error）。
connpool/answer/pool.go
Get 三步逻辑：① 非阻塞取 idle（有空闲连接立即返回）→ ② idle 空则 CAS 新建（total 未达上限）→ ③ 达到上限则阻塞 select 等 idle/closeCh/ctx。每步都检查健康和超时，不合格则 total-- 重新循环。

CAS 的必要性：若用 mutex 保护 total 的读-改-写，则每次 Get 都需要加锁，高并发下成为瓶颈。CAS 失败代价小（重新循环），在连接数未满时几乎总是成功一次。

Close 顺序严格有序：① atomic.Store(closed, 1)（后续 Get 直接返回错误）→ ② close(closeCh)（唤醒阻塞的 Get）→ ③ inUseDone.Wait()（等借出连接归还）→ ④ 排空 idle。顺序不能乱，否则可能泄漏连接或 double-close。

核心考察点

问题答案要点
idle channel 为何能当信号量？cap(idle)=maxSize，写满后阻塞相当于 semaphore；但这里 total 独立计数，channel 只放空闲连接，借出的不在 channel 里。
为何 Put 用非阻塞 select+default？极端情况（如 Close 后仍有连接归还），idle 已满则 Put 阻塞。non-blocking 防止归还方永久阻塞。
如何防止 Close 时泄漏借出的连接？inUseDone WaitGroup：每次 Get 成功 Add(1)，Put 时 Done()；Close 调用 Wait() 阻塞直到全部归还。
健康检查在哪里做？惰性：取出时检查（Get）和归还时检查（Put），不健康直接丢弃并 total--。无后台 goroutine，简单可靠。

限流器 流量控制

令牌桶（惰性补充）+ 滑动窗口（时间戳队列），两种经典限流算法完整实现。

设计要点

  • 令牌桶 — 惰性补充：不起后台 goroutine，每次 Allow/Wait 时按"经过时间 × rate"计算补充量，min(tokens+补充, burst)
  • Allow 非阻塞；Wait 计算需要等待的时长，time.After + ctx 双路 select
  • 滑动窗口 — 时间戳队列：维护升序时间戳切片，每次 Allow 先淘汰窗口外记录，再判总数是否达到 limit
  • 滑动窗口空间复杂度 O(limit)：超过 limit 的请求被拒，不追加时间戳，无内存泄漏
  • 两者都用 sync.Mutex，全程持锁，并发安全
ratelimiter/answer/ratelimiter.go
令牌桶惰性补充的优势：无后台 goroutine，无需 Stop/Close，生命周期简单。代价是每次操作多一次时间计算（纳秒级，可忽略）。Go 标准库 golang.org/x/time/rate 也采用这种设计。

Wait 的原子性：先释放锁计算等待时长，等待结束后再次加锁 refill+扣令牌。两次加锁之间令牌可能被其他 goroutine 消耗，所以等待结束后必须再次 refill，确保令牌充足。

滑动窗口 vs 固定窗口：固定窗口（计数器在整点重置）在窗口边界有"突刺"问题——前一窗口末尾 + 后一窗口开头可以瞬间通过 2×limit 请求。滑动窗口以"当前时刻往前推 windowSize"为视口，无突刺，但内存占用更高。

核心考察点

问题答案要点
令牌桶 vs 漏桶？令牌桶允许突发（桶满时可消耗 burst 个令牌）；漏桶以恒定速率流出，无突发能力。令牌桶更适合允许短暂流量峰值的场景。
滑动窗口的时间复杂度？O(k)，k 为窗口内请求数。淘汰过期记录是线性扫描；若用二分搜索可优化到 O(log k)，但窗口通常不大，线性已足够。
为何滑动窗口空间是 O(limit)？超过 limit 的请求被拒绝不追加，时间戳切片最多 limit 个元素；旧时间戳被淘汰后切片缩减，内存稳定。
生产中用哪个？单机限流优先令牌桶（x/time/rate）；分布式限流用 Redis INCR + expire（固定窗口）或 Redis sorted set（滑动窗口）。

1 过桥问题 — 4 人 1 手电
题目

4 人夜晚过独木桥，每次最多 2 人，只有 1 只手电，必须有人持电筒才能过桥（不能扔回）。4 人单独过桥时间分别为 1、2、5、10 分钟，两人同行取较慢者。最少几分钟全部过桥？

关键思路

瓶颈是 5 和 10 这两个"慢人"，他们必须一起过桥，这样 10 分钟只被计一次。让最快的 1 和 2 专门"跑腿"送电筒回来。

步骤动作耗时此岸对岸
①1、2 同行过桥2 min5、101、2 🔦
②1 返回1 min1、5、10 🔦2
③5、10 同行过桥10 min1 🔦2、5、10
④2 返回2 min1、2 🔦5、10
⑤1、2 同行过桥2 min—全部 ✓
答案：17 分钟
误区：让 1 每次陪人过，耗时 1+10+1+5+2 = 19 min。关键是让两个慢人结伴，让两个快人轮流回送电筒。
2 称球问题 — 12 球 3 次天平
题目

12 个外观相同的球，其中 1 个是异常球（可能偏轻或偏重，未知），其余 11 个重量相同。只有一架无砝码天平，最多称 3 次，找出异常球并确定其轻重。

关键思路

3 次天平有 3³ = 27 种结果，需区分 12×2 = 24 种情况，信息量够用但不能浪费。

  • 分组：A(1–4)、B(5–8)、C(9–12)
  • 第 1 次：A vs B
    • 平衡 → 异常球在 C，再 2 次可定位（已有正常球对照）
    • 不平衡（设 A 重）→ 异常球要么在 A（偏重），要么在 B（偏轻）
  • 第 2 次（A 重情况）：取 A 中 3 枚 + B 中 3 枚 vs C 中 3 枚 + 正常球 3 枚（混洗法），根据倾斜方向缩小候选
  • 第 3 次：在 3 球候选中 1 vs 1 确认
答案：3 次必然找到异常球，并确定其轻重
核心：不能单纯二分，因为每次天平结果有三种（左重/右重/平衡），需要设计"混洗"称量方案来同时利用三种结果。
3 百人囚犯找号码 — 循环策略
题目

100 名囚犯编号 1–100。狱卒将 1–100 的号码随机打乱放入 100 个盒子（每盒一号码）。囚犯依次单独进入房间，每人最多打开 50 个盒子，找到自己编号才算成功。所有人成功则全部释放，一人失败则全部处决。囚犯可以事先商量策略，进房间后不能留记录也不能通信。有没有让全部人成功概率 > 30% 的策略？

关键思路：跟踪循环

100 个盒子和 100 个号码构成一个置换，任何置换都可分解为若干不相交的循环。

策略：编号为 i 的囚犯，先开盒子 i，读出号码 j，再开盒子 j，如此沿循环追踪，直到找到号码 i 或已开 50 个。

27 轮流抛硬币 — 先手赢的概率
题目

两人轮流抛一枚公平硬币（正面概率 1/2），先抛到正面者赢。先手的人赢的概率是多少？

关键思路

设先手赢的概率为 P。

  • 先手第 1 次抛出正面（概率 1/2）→ 先手赢
  • 先手反面（1/2）且后手也反面（1/2）→ 回到初始状态，先手赢概率仍为 P

列方程：P = 1/2 + (1/4)P → (3/4)P = 1/2 → P = 2/3

验证（等比数列）：1/2 + (1/2)³ + (1/2)⁵ + … = (1/2) / (1 − 1/4) = 2/3 ✓

答案：先手赢的概率为 2/3，后手为 1/3
先手优势来自"第一次机会"。推广：若硬币正面概率为 p，先手赢概率 = p / (1−(1−p)²)。当 p=1/2 时 = 2/3。

  • B 负责先选，选走他认为更大的那份

  • 因数个数为奇数 ⟺ n 是完全平方数

B 猎鹿B 猎兔
A 猎鹿(5, 5) ✓(0, 3)

58 重复囚徒困境 — 以牙还牙策略
题目

囚徒困境（合作各+3，背叛+5/合作者0，双背叛各+1）无限重复。能否维持合作？什么是以牙还牙策略（TFT）？

关键思路

以牙还牙（Tit-for-Tat）：第 1 轮合作，之后每轮复制对方上一轮行为。

特点含义
善良（Nice）不先背叛
报复（Retaliatory）立即惩罚背叛
宽容（Forgiving）对方恢复合作后立即原谅
清晰（Clear）策略简单，对手容易理解

维持合作的条件：折现因子 δ > 1/3（背叛额外收益 2 < 未来合作损失 2δ/(1-δ)）。

答案：δ>1/3 时 TFT 可维持合作均衡。Axelrod 计算机锦标赛中 TFT 击败所有复杂策略
TFT 的局限：在有噪声的环境中（偶尔误判对方行为），TFT 会陷入"报复循环"。改进版"慷慨的 TFT"（偶尔原谅背叛）更稳健。
59 性别战博弈 — 混合策略均衡计算
题目

夫妻选择拳击（夫偏爱）或芭蕾（妻偏爱），但更想在一起。一起拳击(夫2,妻1)，一起芭蕾(夫1,妻2)，分开(0,0)。找出所有均衡，混合均衡下在一起的概率是多少？

关键思路
妻:拳击妻:芭蕾
夫:拳击(2,1) ✓(0,0)
夫:芭蕾(0,0)(1,2) ✓

混合均衡：设夫以概率 p 选拳击，妻以概率 q 选拳击。

  • 妻无差异：2q = 1-q → q = 1/3
  • 夫无差异：2p = 1-p → p = 2/3

在一起概率 = pq + (1-p)(1-q) = (2/3)(1/3) + (1/3)(2/3) = 4/9 ≈ 44%

答案：3 个均衡（拳击拳击、芭蕾芭蕾、混合策略）。混合均衡下在一起概率 4/9，低于纯策略均衡的 100%

1 设计推特（Twitter Feed）
题目

设计一个类似 Twitter 的信息流系统。用户可以发布推文、关注其他用户，并查看关注者的最新时间线（Home Timeline）。要求支持海量用户（亿级）、高并发读（每秒百万级）。

设计要点
  • 规模估算：日活 1 亿，每用户平均关注 200 人，发推 QPS ~5000，读 Timeline QPS ~50 万
  • 核心难点：写扩散 vs 读扩散。大 V（百万粉）发一条推文，若写扩散需写入百万用户的 Timeline 缓存
  • Fanout 策略：普通用户（<1 万粉）写时扇出到粉丝 Feed 缓存；大 V 走读时合并（避免写放大）
  • 存储：推文表用 MySQL（按 user_id 分片）；Timeline 缓存用 Redis Sorted Set（score=时间戳）
  • 消息队列：发推事件投入 Kafka，消费者异步写扩散，削峰解耦
  • CDN：媒体文件（图片/视频）通过 CDN 分发，减少源站压力
推荐方案
用户发推 → 写 MySQL（推文持久化） → 投 Kafka（topic: new_tweet） → 消费者判断发送者粉丝数 ≤ 1万 → 遍历粉丝，写 Redis ZADD timeline:{uid} score=ts tweet_id > 1万 → 只写推文，不扇出（大V走读时合并） 用户读 Timeline → Redis ZREVRANGE timeline:{uid} 0 99 → 大V 的推文：单独查 MySQL，合并排序 → 批量查推文详情（多级缓存）
层次技术选型说明
推文存储MySQL（按 user_id 分片）持久化，支持复杂查询
Timeline 缓存Redis Sorted Set高速读，score=时间戳
消息队列Kafka异步扇出，削峰
媒体存储对象存储 + CDN图片/视频分发
搜索Elasticsearch推文全文检索
关键权衡：写扩散（降低读延迟）vs 读扩散（降低写放大）。实践中通常混合使用：普通用户写扩散，大 V 读扩散合并。
2 设计 URL 短链服务
题目

设计一个 URL 缩短服务（类似 bit.ly）。将长 URL 映射为 6~8 位短码，访问短链时 302 重定向到原始 URL。要求：写 QPS ~1000，读 QPS ~10 万，短链有效期可配置。

设计要点
  • 短码生成：62 进制（a-z A-Z 0-9），6 位 = 56 亿种组合，满足需求
  • 发号器方案：数据库自增 ID 转 62 进制（推荐，无冲突，号段模式减少 DB 访问）
  • 哈希方案：MurmurHash 取前 6 字节转 62 进制，冲突时自增重试
  • 重定向：302（临时，不缓存，统计准确）vs 301（永久，客户端缓存，减少服务器压力）
  • 缓存：热点短链 Redis String 缓存，TTL 与短链一致
  • 去重：同一长 URL 可复用已有短码（Redis 缓存长→短的映射）
推荐方案
写请求（长 URL → 短码） → 检查 Redis：该长URL是否已有短码（去重） → 无缓存：向发号器服务申请自增 ID（号段模式） → ID 转 62 进制得到短码 → MySQL INSERT (short_code, long_url, expire_at) → Redis SET short:{code} long_url EX ttl → 返回短链 读请求（短码 → 重定向） → Redis GET short:{code} → 未命中 → MySQL 查询 → 写回 Redis → 找到 → HTTP 302 Location: long_url
组件技术说明
发号器MySQL auto_increment + 号段模式每次领取 1000 个 ID，减少 DB 访问
持久化MySQL（short_code 主键）唯一约束防冲突
缓存Redis String热点短链，减少 DB 读压力
统计Kafka + ClickHouse异步统计点击量、来源
关键决策：302 vs 301。302 每次都回源服务器，统计数据准确但压力大；301 浏览器缓存后不再回源，省流量但统计不准。面试中选 302 + 服务端缓存是最常见答案。
3 设计微信群聊消息系统
题目

设计微信群聊消息系统。支持万人群，消息实时推送，离线消息存储，消息序号保证有序，支持已读/未读状态。

设计要点
  • 消息序号（msg_id）：每个群维护单调递增序号（Redis INCR 原子操作），保证有序
  • 万人群写放大：不为每个成员写一条收件记录（写放大 1 万倍），改用群消息 + 成员游标（cursor）
  • 离线消息：用户上线后，查询各群 cursor 之后的消息，批量拉取
  • 推送模型：在线用户 WebSocket 实时推送；离线用户上线 pull；接入层 push-server 一致性哈希路由
  • 已读回执：更新 cursor 即视为已读，未读数 = 群最新 msg_id - cursor
  • 消息存储：按 group_id 分片 MySQL，冷数据归档对象存储
推荐方案
发消息 → Redis INCR group:{gid}:seq 获取 msg_id → 写 MySQL msg 表（gid, msg_id, sender, content, ts） → 投 Kafka（topic: group_msg） → 消费者：对在线成员 WebSocket 推送 msg_id 用户上线 → 查 cursor 表：获取各群最后已读 msg_id → SELECT * FROM msg WHERE gid=? AND msg_id > cursor LIMIT 100 → 推送离线消息，更新 cursor
问题方案
消息有序群级别自增 msg_id（Redis INCR 原子）
万人群写放大不写扩散，成员维护 cursor，上线 pull
实时推送WebSocket 长连接，接入层一致性哈希路由
未读数群最新 seq - cursor，Redis 缓存
关键权衡：推（push）vs 拉（pull）。万人群不做写扩散，改为 cursor + pull，是解决写放大的核心思路。实时性靠 WebSocket 推 msg_id，内容靠 pull 补全。
4 设计秒杀系统
题目

设计一个双十一秒杀系统。1000 件商品，100 万用户同时抢购，要求不超卖、不少卖、高可用，系统 QPS 峰值可达 10 万。

设计要点
  • 流量漏斗：请求在到达数据库之前应被大量过滤，只有极少数能到扣库存环节
  • 限流：Nginx 层令牌桶限流 → 网关层用户级限频（同一用户 1 秒只处理 1 次）→ MQ 削峰
  • Redis 扣库存：Lua 脚本原子执行"判断库存 > 0 + DECR"，避免 ABA 问题
  • 幂等性：Redis SET 用户+商品唯一键（NX），防止同一用户重复购买
  • 异步下单：扣库存成功后投 Kafka，消费者异步写 DB；用户轮询订单状态
  • DB 兜底：消费者写 DB 时用乐观锁（WHERE stock > 0）再次保证不超卖
推荐方案
请求链路： 用户 → CDN（静态资源缓存，不回源） → Nginx 限流（令牌桶，10w→1w QPS） → API Gateway（用户级限频、鉴权） → 秒杀服务 1. Redis SET seckill:uid:gid 1 NX EX 3 （去重） 2. Lua: if stock>0 then DECR stock end （原子扣库存） → 失败 → 返回"已售罄" 3. 投 Kafka（topic: seckill_order） 4. 返回"排队中" 消费者： → 从 Kafka 消费 → MySQL INSERT order + UPDATE stock WHERE stock>0（乐观锁） → 写订单状态到 Redis → 用户轮询 GET /order/status
问题方案原因
超卖Redis Lua 原子扣减 + 乐观锁兜底原子操作 + DB 最终校验
重复购买Redis NX 去重键原子 SET，TTL 自动清理
流量削峰Kafka 缓冲 + 消费者异步写 DB解耦，保护 DB
关键思路：流量漏斗 + 原子扣减 + 异步化。库存超卖的本质是非原子的"读-判断-写"，Redis Lua 脚本将这三步合并为原子操作是标准解法。
5 设计消息队列（简化版 Kafka）
题目

设计一个简化版消息队列系统，支持：Producer 发布消息、Consumer 订阅消费、消息持久化、至少一次投递（at-least-once）、消费者组（Consumer Group）负载均衡。

设计要点
  • 核心抽象：Topic → Partition（分区）→ Segment（分段日志文件）
  • 顺序写磁盘：消息追加写到 Partition 日志（Append-Only），顺序 I/O 接近内存速度
  • Offset：消费者维护消费位点，重启后从上次 offset 继续，实现 at-least-once
  • Consumer Group：同一 Group 内，一个 Partition 只被一个 Consumer 消费，保证组内不重复消费
  • 副本（Replication）：每个 Partition 有 N 个副本，Leader 接收读写，Follower 同步；ISR 机制保证持久化
  • 稀疏索引：每隔 4KB 记录一个 offset→文件位置，加速随机读
推荐方案
存储结构： /data/topic-name/ partition-0/ 00000000.log ← 消息文件（顺序追加） 00000000.index ← 稀疏 offset 索引 partition-1/ ... 写入流程： Producer → 按 key hash 选 Partition（无 key 则轮询） → 发送到 Partition Leader → Leader 追加写 .log → ISR 内 Follower 同步确认后 ack 消费流程： Consumer → 加入 ConsumerGroup，Coordinator 分配 Partition → 从 committed offset 开始 Fetch → 处理成功 → Commit offset → 失败 → 不 commit，下次重新投递（at-least-once）
特性实现
高吞吐顺序写磁盘 + 零拷贝（sendfile）
持久化Append-Only 日志，多副本 ISR
有序同一 Partition 内严格有序
负载均衡Partition 分配给 Consumer Group 成员
容灾ISR 副本 + Leader 自动选举
关键设计：顺序写 + Partition 并行是 Kafka 高吞吐的核心。Consumer Offset 由消费者自己维护，使得重放（replay）和 at-least-once 投递天然支持。
6 设计分布式 ID 生成器
题目

设计一个分布式 ID 生成服务，满足：全局唯一、趋势递增（有利于 B+ 树索引写入）、高性能（单机 >10 万 QPS）、高可用（无单点）、ID 不暴露业务信息（不可被枚举）。

方案对比
方案唯一性趋势递增性能主要缺点
UUID v4 ✅ 全局唯一 ❌ 完全随机 ✅ 本地生成，极快 128 bit，字符串存储大；随机写导致 B+ 树页分裂频繁，写性能差；不含时间，排序无意义
DB 自增 ID ✅（单库） ✅ 严格递增 ❌ 受 DB 并发限制 单点瓶颈；多库步长方案（每台步长 N、起始不同）可扩展，但扩容麻烦，ID 暴露分库规模

Snowflake ✅ ✅ 趋势递增 ✅ 本地生成，极快 依赖系统时钟；时钟回拨会导致 ID 重复（核心问题）；workerId 需外部分配
美团 Leaf ✅ ✅ ✅ 号段缓存，内存发号 号段模式：DB 宕机时缓存内 ID 可继续使用；Snowflake 模式：用 ZK 分配 workerId，解决时钟回拨
百度 UidGenerator ✅ ✅ ✅ RingBuffer 预生成 基于 Snowflake 变体，workerId 用 DB 分配（应用启动注册），RingBuffer 异步填充消除毛刺
Snowflake 结构详解
64 bit ID 结构： ┌─────────┬──────────────────────────────────────┬──────────────┬────────────────┐ │ 符号位 │ 时间戳（毫秒） │ workerId │ 序列号 │ │ 1 bit │ 41 bit │ 10 bit │ 12 bit │ └─────────┴──────────────────────────────────────┴──────────────┴────────────────┘ ↑ ↑ 最多 1024 台机器 同毫秒内最多 4096 个 ID • 41 bit 时间戳：相对自定义 epoch（如 2020-01-01），可用约 69 年 • 10 bit workerId：5 bit datacenter + 5 bit worker，共 1024 个节点 • 12 bit 序列号：同一毫秒内自增，超过 4095 则等待下一毫秒 • 单机 QPS 上限：1000ms × 4096 = 约 409.6 万/秒
时钟回拨问题与解决方案
  • 问题根因：NTP 时间同步可能让系统时钟往回跳，导致生成的 ID 时间戳小于上次 ID，破坏唯一性
  • 方案 1 — 拒绝服务：检测到回拨直接抛错（回拨 < 2ms 时等待追上，> 2ms 直接报警）。简单但影响可用性
  • 方案 2 — 扩展位借用：将 workerId 的高位几 bit 作为"时钟回拨次数"计数器，每次回拨自增该计数，保证 ID 仍全局唯一（美团 Leaf Snowflake 思路）
  • 方案 3 — 使用逻辑时钟：不依赖系统时钟，改用单调递增的逻辑时钟（如 ZooKeeper 的 zxid），彻底消除回拨风险
  • 方案 4 — 号段模式兜底：Snowflake 出问题时降级为 DB 号段模式（双 buffer，提前预取），保证高可用
美团 Leaf 双模式
模式原理特点
号段模式（Segment） DB 表维护每个 bizTag 的 max_id 和 step；服务批量取号（如一次取 1000 个），内存中依次发放；剩余量低于 10% 时异步预取下一段 DB 宕机后内存中仍有号可用；双 buffer 消除取号延迟毛刺；ID 连续但会有跳跃（重启丢失当前号段剩余量）
Snowflake 模式 用 ZooKeeper 持久节点存储 workerId，服务启动时注册获取，并定期上报心跳时间戳 ZK 宕机时本地缓存 workerId 可继续运行；利用上报的时间戳检测时钟回拨，回拨超阈值则拒绝发号
推荐方案（生产选型）
场景一：中小规模，追求简单 → Redis INCR + 号段批量取（每次取 1000），应用内存顺序发放 → 优点：无需额外服务，接入成本低 → 注意：AOF 持久化 + 每次取号后立即持久化（fsync）防丢失 场景二：大规模，追求极致性能 → 部署美团 Leaf（号段模式 + Snowflake 模式双保险） → 号段模式作为主力，Snowflake 作为高性能补充 → workerId 通过 ZK 分配，规避手动管理 场景三：容器 / K8s 环境，动态扩缩容 → 百度 UidGenerator：DB 注册 workerId（Pod 启动时写 DB 获取唯一 workerId） → RingBuffer 异步预生成 ID，填充率低时后台线程补充，彻底消除阻塞 通用原则： 1. 业务 ID 不要直接用 DB 自增主键对外暴露（可被枚举订单量） 2. 趋势递增 > 严格递增（允许同毫秒内乱序，只要整体趋势递增即可） 3. 时钟回拨处理：小回拨等待，大回拨降级到号段模式
核心考点：Snowflake 的 64 bit 拆分结构（1+41+10+12）、时钟回拨的三种解法（等待/借位/逻辑时钟）、Leaf 号段模式的双 buffer 设计。面试中能完整画出 Snowflake bit 布局 + 说清楚时钟回拨解法，基本能拿满分。
7 设计 Feed 流系统

设计一个通用 Feed 流系统（类微博/朋友圈/抖音）。用户关注其他用户后，能在首页看到关注者按时间倒序排列的内容流。要求：日活 1 亿，写 QPS ~5 万，读 QPS ~50 万，支持普通用户和大 V（千万粉），延迟 <200ms。

  • 订阅（Subscribe）：关注关系存储，关注者能看到被关注者的内容

sync.Map 并发 · 原子操作

从零实现 sync.Map：两张表 + miss 提升机制 + expunged 哨兵，彻底理解标准库设计。

设计要点

  • 两张表：read（atomic.Value，无锁读）+ dirty（mutex 保护，全量数据）
  • fast path：Load 先查 read，命中则无需加锁，适合读多写少场景
  • miss 提升：每次 read 未命中计 misses；当 misses >= len(dirty) 时将 dirty O(1) 提升为 read，之后读再次走 fast path
  • expunged 哨兵：dirtyLocked 重建 dirty 时，将 read 中 p==nil（软删除）的 entry 标记为 expunged，不拷入 dirty；后续对该 key 的 Store 通过 unexpungeLocked 恢复
  • entry.p 三态：正常指针（存在）/ nil（软删除）/ expunged（从 read 删除且不在 dirty）
  • 所有 entry 方法用 CAS（无锁）；只有 dirty 操作需要 mutex
syncmap/answer/syncmap.go
为何需要两张表：普通 map + RWMutex 在高并发读时，多个 goroutine 争抢 RLock 仍有竞争开销。sync.Map 通过 atomic.Value 存储 read 快照，读操作完全无锁——只有 read 未命中时才加锁查 dirty。

expunged 为何必要：考虑以下序列：① Store(k,1) → k 在 read；② 触发 dirtyLocked 重建 dirty；③ Delete(k) → read 中 entry.p=nil；④ Store(k,2)。步骤④时，若不区分 nil 和 expunged，就无法判断该 entry 是否需要重新加入 dirty。expunged 明确表示"dirty 中没有我"，Store 发现后通过 unexpungeLocked 把 entry 重新插回 dirty。

miss 提升时机：每次走 dirty 路径都累计 misses。当 misses >= len(dirty) 时，说明 dirty 的每个 key 都被查过了，未来大概率还会被查，此时将 dirty 整体替换 read（仅交换指针，O(1)），reset dirty=nil，misses=0。

double-check 模式：Load/Store 的 slow path 加锁后需要重新 loadRead()——因为加锁等待期间另一个 goroutine 可能已完成 dirty 提升，若不重新检查会遗漏 fast path 机会或操作过时的 dirty 引用。

核心考察点

问题答案要点
sync.Map 适合哪些场景？不适合哪些？适合：读多写少（缓存）、不同 goroutine 操作不相交的 key。不适合：频繁写同一批 key——dirty 反复重建，miss 提升开销大，不如 map + RWMutex。
read 和 dirty 的关系？dirty 是全量数据（包含 read 中没有的新 key）；read 是 dirty 的子集快照。提升后 dirty=nil，read 成为全量。二者共享同一批 *entry 指针，写 entry 时无需同步两张表。
expunged 和 nil 的区别？nil：软删除，dirty 中已无此 key，但 read 残留。expunged：dirty 重建时标记，表示"dirty 中从未有过我"，后续 Store 需通过 unexpungeLocked 恢复。没有 expunged 就无法区分"删除前未入 dirty"和"删除后已从 dirty 移除"。
为何 Load 里要 double-check？获取 mu 前 read 快照可能已过期（另一个 goroutine 完成了提升），不 double-check 会在旧的 dirty 上操作，甚至 dirty==nil 时 panic。
Range 为何不会死锁？Range 先（短暂加锁）将 dirty 提升为 read，然后在无锁下遍历 read 快照。f 内调用 Load/Store/Delete 可安全加锁，因为 Range 自己已释放锁。

String 类 字符串 · 算法

实现不可变 MyString 类型：KMP 搜索、UTF-8 感知反转、高效 Builder 拼接。

设计要点

  • 不可变性：内部 []byte，所有方法返回新实例；Bytes() 做防御性拷贝，不暴露内部引用
  • KMP 搜索：Index/Contains 时间复杂度 O(n+m)；核心是 failure function（部分匹配表），失配时回退 pattern 指针而非文本指针
  • UTF-8 感知：Reverse（字节反转，ASCII 安全）vs ReverseRunes（先解码 []rune，交换后重新编码）
  • Builder 模式：用 append([]byte) 摊销内存分配，O(n) 拼接；相比 += 的 O(n²) 有本质改进
  • 链式调用：b.Write(...).WriteString(...).WriteByte(...)，每个方法返回 *Builder

不可变性实现：MyString.b 是未导出字段，外部无法访问。Bytes() 必须返回拷贝——若直接返回 s.b，调用方持有切片引用后修改，将静默破坏内部状态。Concat/Slice 同理，均创建新的底层数组。

KMP failure function：f[i] = pattern[0..i] 最长公共真前缀后缀的长度。例如 "ababc" → [0,0,1,2,0]。搜索时失配发生在位置 j，说明 pattern[0..j-1] 已匹配，利用 f[j-1] 跳过不可能成功的起点，文本指针 i 永不回退。

字节反转 vs rune 反转：对 "你好"（UTF-8 编码为 e4 bd a0 e5 a5 bd）直接字节反转得到 bd a5 e5 a0 bd e4，是无效 UTF-8。ReverseRunes 先 []rune("你好") 解码为 [0x4F60, 0x597D]，交换后重新编码，结果正确。

Builder 原理：append 在容量不足时按 2x 扩容，n 次 append 的总分配次数是 O(log n)，总复制字节数是 O(n)。而 s += t 每次都分配新数组并复制全部已有内容，n 次累计 O(n²)。

核心考察点

问题答案要点
KMP 比暴力算法快在哪里？暴力每次失配都回退文本指针，最坏 O(n*m)。KMP 利用 failure function 跳过已知不匹配的位置，文本指针只前进不回退，O(n+m)。
failure function 的含义？f[i] = pattern[0..i] 中，最长的"既是真前缀又是真后缀"的字符串长度。失配时利用它跳转：j = f[j-1]，表示已匹配的部分中有一个更短的公共前后缀可以继续尝试。
为何 string += 是 O(n²)？Go string 不可变，+= 每次创建新字符串并复制全部内容。n 次操作拷贝 1+2+...+n = O(n²) 字节。
Reverse 和 ReverseRunes 的区别？Reverse 直接逐字节倒序，对多字节 UTF-8 字符会产生非法序列。ReverseRunes 先解码为 []rune，交换的是完整字符，结果始终是合法 UTF-8。
